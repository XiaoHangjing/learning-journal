## Content Insights

The main thing I learned this week was why a child theme is better than editing a parent theme directly.

A child theme inherits the layout and functions of the parent theme, but keeps the custom files separate. This means that the parent theme can still be updated without replacing the CSS changes made for the website. Before this practical, I mostly thought about whether a design worked on the page. I did not think much about whether the changes would still be there after a theme update.

I learned that the `Template` value inside `style.css` must match the exact folder name of the parent theme. In my case, the parent theme was Twenty Twenty-Four, so I used:

```css
Template: twentytwentyfour
```

I also learned that `functions.php` is used to load the child theme stylesheet. This was a small introduction to PHP for me. I did not write any complicated PHP, but I now understand why CSS should be loaded through WordPress instead of adding a stylesheet link manually.

Another useful part of this week was learning more about CSS specificity. When I changed the product cards from a dark background to white, the product names and prices appeared to disappear. The content was still there, but WordPress had already applied a white text colour to those blocks.

At first, I kept adding more colour rules. Some of them did not work, and the CSS file started becoming longer than necessary. The problem was not the colour value. The problem was that my selector was not specific enough to override the WordPress block style.

The selector that finally worked was:

```css
.wp-block-columns .wp-block-column h2,
.wp-block-columns .wp-block-column .wp-block-heading {
    color: #173b3d !important;
    -webkit-text-fill-color: #173b3d !important;
    opacity: 1 !important;
    visibility: visible !important;
}
```

This helped me understand that CSS troubleshooting should not be based on guessing. It is better to inspect the element, check its class and then write a rule that targets the correct part of the page.

I also learned that CSS variables can make a stylesheet easier to manage. For example:

```css
:root {
    --sb-primary: #168c91;
    --sb-heading: #173b3d;
    --sb-background: #f7faf9;
}
```

Instead of repeating the same colour code in many places, I can use `var(--sb-primary)`. If I change the variable later, every element using that variable will update as well.


## Career/Employability/Learning Insights

This practical helped me understand the importance of testing changes carefully and keeping my code organised. Making small adjustments and checking the result each time made it easier to identify problems and understand how different CSS rules affected the page. It also reminded me to review my code regularly and remove anything unnecessary.

I also realised that working with WordPress involves more than changing the appearance of a website. Understanding how themes, stylesheets and different files work together is important for making effective customisations. The skills used in this practical are also relevant to many areas of web development because content management systems are widely used by organisations and businesses.

One area I still need to improve is my understanding of PHP and the files that control WordPress functionality. I was able to follow the examples used in this practical, but I need more practice before I can confidently make more advanced changes independently. My next step is to learn basic PHP syntax in WordPress, especially functions, variables, hooks and template files, and to use the browser Inspector earlier when troubleshooting. Writing a LinkedIn post for this practical was also useful because it made me think more carefully about how to explain my work clearly to people outside the classroom.
